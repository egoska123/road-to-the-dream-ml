# Project: CatBoost Binary Classifier — аномальное потребление ТНС энерго

## Goal

Определение **аномального (коммерческого) потребления** электроэнергии по данным ТНС энерго: бинарная классификация целевого поля `isCommercial`. Задача — по признакам объекта (тип здания, комнаты, жильцы, помесячное потребление и т.д.) предсказать, является ли потребление коммерческим (True) или бытовым (False), что позволяет выявлять случаи некорректной классификации и потенциальные аномалии в учёте.

## Task Type

- **binary classification** (целевой признак: `isCommercial`)

## Dataset / Data Source

- **Источник:** датасеты ТНС энерго по потреблению электроэнергии.
- **Файл:** `dataset_train.json` — обучающая выборка в формате JSON (список словарей).
- **Объём:** 4826 записей.
- **Признаки:**
  - `accountId` — идентификатор лицевого счёта (не используется как признак).
  - `address` — адрес (исключён из модели).
  - `buildingType` — тип здания (категориальный, например «Частный»).
  - `roomsCount`, `residentsCount` — число комнат и жильцов (float, есть пропуски).
  - `consumption_1` … `consumption_12` — потребление по месяцам (кВт·ч), возможны пропуски.
  - `totalArea` — общая площадь (м²), много пропусков.
- **Целевая переменная:** `isCommercial` (bool). Распределение: **False — 2935**, **True — 1891** (дисбаланс классов).
- **Пропуски:** в `roomsCount`, `residentsCount`, в части помесячных потреблений и в `totalArea`; учёт пропусков оставлен на совести CatBoost (нативная работа с NaN).

## Approach

- **Preprocessing:** загрузка JSON через `pd.json_normalize`, переименование полей `consumption.<N>` → `consumption_<N>`. Формирование матрицы признаков без `accountId`, `address` и `isCommercial`. Категориальный признак: `buildingType`.
- **Baseline:** отдельный baseline не описан; по сути baseline — первая версия CatBoost без дополнительных признаков.
- **Main model:** CatBoostClassifier: `iterations=3000`, `learning_rate=0.03`, `depth=5`, `l2_leaf_reg=20`, `auto_class_weights="Balanced"`, `loss_function="Logloss"`, `eval_metric="F1"`, early stopping по validation F1 (`od_type="Iter"`, `od_wait=100`). Обучение останавливается на итерации 508 (best F1 на valid).
- **Evaluation:** разбиение 70% train / 15% valid / 15% test со стратификацией по `isCommercial`. На validation перебираются пороги от 0.1 до 0.9 с шагом 0.05; для теста используется порог 0.5 (оптимальный по F1 на valid).
- **Comparison:** сравнение метрик при разных порогах зафиксировано в ноутбуке (F1/precision/recall по valid); финальная оценка — по test.

## Project Structure

- `tns-energo-ml.ipynb` — основной ноутбук: загрузка данных, препроцессинг, обучение CatBoost, подбор порога, оценка на test.

Отдельных папок `notebooks/`, `src/`, `results/`, `reports/` в проекте нет; весь код и эксперименты сосредоточены в одном ноутбуке.

## What worked

- CatBoost с нативной поддержкой категориальных признаков (`buildingType`) и пропусков.
- `auto_class_weights="Balanced"` для учёта дисбаланса классов (меньшинство — коммерческое потребление).
- Early stopping по F1 на validation — обучение останавливается на 508 итерации, снижается риск переобучения.
- Подбор порога по F1 на validation: порог 0.5 даёт разумный баланс precision/recall на test.

## What did not work

- Дополнительные признаки из функции `add_features` (пропуски как индикаторы, агрегаты по потреблению, сезонность, нормировки) в текущем пайплайне **не используются** — вызовы закомментированы; их влияние на метрики не оценивалось.
- Опечатка в коде: переменная `categorial_features` вместо `categorical_features` в print (не ломает обучение, т.к. в `fit` передаётся отдельно определённый список).

## Main takeaway

Модель уверенно отделяет часть коммерческих объектов от бытовых (F1 по классу True ~0.59, accuracy 0.66), но класс «коммерческое потребление» предсказывается хуже из‑за дисбаланса и, возможно, недостатка информативных признаков. Имеет смысл включить отложенные фичи из `add_features`, посчитать ROC-AUC и при необходимости настроить порог под задачу (например, выше recall по коммерческому классу).
